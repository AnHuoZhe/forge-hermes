# 锻造核心流程定义

本文件定义锻造的核心流程和标准。Codex版和Hermes版共用此定义。平台特定的实现细节（dispatch subagent vs delegate_task）在各版本SKILL.md中处理。

---

## 五阶段流程

### 阶段A：需求澄清

目标：把模糊想法变成明确需求文档。

必问问题：
- 要解决什么问题？
- 核心功能3-5项
- 明确不做的事
- 技术约束（语言、平台、部署）
- 目标用户是谁？

产出文件：`forge-brief.md`

```
# 项目名称 - 需求文档

## 项目目标
一句话

## 核心功能
- 功能1：xxx

## 明确不做
- xxx

## 技术约束
xxx

## 目标用户
xxx
```

确认后进入阶段B。

### 阶段B：架构设计

目标：基于需求文档设计技术架构。

产出文件：`forge-design.md`

```
# 项目名称 - 架构设计

## 模块划分
- 模块A：职责xxx

## 数据流
xxx

## 接口定义
xxx

## 技术栈
- 语言：xxx
- 选型理由：xxx

## 任务拆解
1. 任务1：xxx（预期产出：xxx）
```

展示全文给用户确认后进入阶段C。

### 阶段C：审查

目标：独立子agent审查设计文档。精审+通审，多轮对撞直到收敛。

**审查层级**：
- 精审 = 架构裂隙分析 + 六维度分析（涉及Agent设计时）
- 通审 = 常规标准收敛

**架构裂隙分析（五层切口）**：
1. 数据流：模块间数据怎么流转？有断点吗？
2. 失败模式：每个环节可能怎么失败？有处理方案吗？
3. 状态切换：系统状态、切换条件、非法转换
4. 分层检查：层间职责清晰？有跨层耦合吗？
5. 运维隐患：部署、日志、监控、备份、配置

**六维度分析（仅涉及Agent/LLM设计时启用）**：
| 维度 | 检查点 |
|------|-------|
| 上下文工程 | 信息注入时机和分层 |
| 记忆系统 | 状态持久化和跨会话记忆 |
| 工具调用 | 工具定义能否让LLM正确选择 |
| 可靠性 | 输出校验和重试机制 |
| 成本控制 | token消耗可控性 |
| 评估 | 系统是否正常工作的判断标准 |

五个交叉点为核心：上下文×成本、记忆×上下文、工具×可靠性、可靠性×评估、成本×评估。

**收敛规则**：
- 最多5轮
- 通过标准：必须改=0 且 建议改不增 且 自问段无新严重担忧
- 同一问题回退（上轮改好这轮又出）立即终止，标记异常

**审查报告格式**（`forge-review-{N}.md`）：
```
# 审查报告 - 第N轮

## 必须改（不改系统不可用）
1. **问题** / **位置** / **原因** / **建议**

## 建议改（改了更好但不阻塞）
1. **问题** / **建议**

## 可忽略
- xxx

## 自问段
- 判断有漏洞吗？过度审查吗？
```

### 阶段D：TDD逐任务实现

目标：从设计文档的任务列表逐个实现。先测试后实现。

**TDD要求**：
1. 写测试：正常输入、边界条件、错误输入
2. 确认测试失败（红）
3. 写实现：直接改文件，不先出设计说明
4. 确认测试通过（绿）
5. 重构（可选）

**规则**：
- TDD不可跳过，顺序不可反
- 如果发现设计问题，标注但不自行修改设计文档
- 不创建多余文件（README、package.json等，除非任务要求）

**每个任务完成后**：输出git diff确认改动内容。

### 阶段E：归档

产出文件：`forge-summary.md`

```
# 项目总结

## 做了什么
## 关键决策
## 遇到的问题
## 后续方向
```

---

## 通用规则

- git commit格式：中文，"第X版，做了什么"
- 阶段A和B交互式（需用户参与决策）
- 阶段C和D自动派发（干净上下文，子agent独立运行）
- 审查通过后直接进入实现，不反复确认
